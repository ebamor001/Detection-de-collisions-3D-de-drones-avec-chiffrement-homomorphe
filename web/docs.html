Documentation

Comprendre la méthode : CKKS pour la géométrie, comparator TFHE-style pour décider “oui/non”.

Motivation
Pourquoi la confidentialité ?

Les trajectoires peuvent révéler des informations sensibles (missions, zones, habitudes, itinéraires). On veut donc calculer un indicateur de collision sans divulguer les chemins.

Objectif
Ce qui est révélé

Uniquement un résultat minimal : collision / pas collision. Aucun point de trajectoire n’est révélé.

CKKS (calculs réels)

CKKS permet d’effectuer des opérations (additions, multiplications) sur des valeurs réelles chiffrées. C’est adapté aux calculs géométriques (distances, projections, produits scalaires), avec une précision contrôlée.

Comparator TFHE-style (booléens)

Pour décider “collision ?”, il faut souvent une comparaison ou un signe (ex: x > 0). Les comparaisons étant coûteuses en HE approximatif, on utilise un comparator binaire de style TFHE (souvent via scheme switching) pour obtenir un booléen.

Protocole simplifié

1
KeyGen

Bob génère les clés (CKKS + éléments comparator).

2
Chiffrement

Les trajectoires sont encodées/chiffrées.

3
Calcul

Intersection calculée sur données chiffrées.

4
Révélation

Seul le booléen final est révélé à Bob.

Limites / points d’attention

  • Coût supérieur au calcul en clair (temps, mémoire, taille des chiffrés).
  • CKKS est approximatif : gestion des échelles et de la précision.
  • Comparaisons coûteuses → comparator dédié.
